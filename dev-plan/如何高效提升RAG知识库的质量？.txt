# 如何高效提升RAG知识库的质量？

这个问题我太有共鸣了。因为我自己踩过的坑，可能比你还多。

先说一句大实话：如果你按照默认流程搭RAG——文档丢进去、自动切chunk、生成embedding、扔进向量数据库——命中率能到60%就算祖坟冒烟了。

大部分人的RAG效果差，不是因为模型不行、也不是向量数据库不行，而是数据预处理这一步就烂了。垃圾进，垃圾出，后面的环节再怎么优化都是在屎山上盖楼。

下面讲的都是我实际调过、有效的方法。不讲论文，讲工程。


一、先诊断：你的RAG到底烂在哪里
很多人一上来就调参数，但你连问题出在哪都没搞清楚。

RAG的流程只有两个核心环节：检索和生成。效果差，要么是检索没找对，要么是找对了但LLM没答好。两个问题的解法完全不同。

怎么判断？ 做一个简单的实验。

把你的测试问题拿出来，先看检索返回的Top-K结果——不看LLM的回答，只看检索到的原始chunk。 如果Top-K里根本没有包含正确答案的chunk，那问题出在检索端。如果Top-K里有正确答案但LLM没答对，那问题出在生成端。

根据我的经验，80%以上的RAG效果差，根源都在检索端。 所以本文主要讲检索端怎么优化。

二、分段（Chunking）：这是90%的人做错的地方
默认的分段方式是什么？按token数切，比如512token一段，有个overlap。

这是最懒、也是效果最差的方式。

为什么？因为它完全不管语义边界。一个完整的概念可能被切成两半，分到了两个chunk里。用户问的时候，两个半截都匹配不上，答案就丢了。

原则一：按语义切，不要按长度切
你的文档不是一堆无差别的文字，它有结构——标题、段落、小节、表格、列表。按照文档自身的结构来切，效果比按token数切好一个量级。

具体怎么做：

如果是有明确标题层级的文档（比如技术文档、产品说明书），按标题层级切。一个二级标题下的内容就是一个chunk。如果太长就在三级标题处再切一刀。
如果是没有标题的长文本（比如合同、法律条文），用LLM来做语义分段——让它读一遍文档，标出"这里话题变了"的位置，按这些位置切。
表格单独处理。 不要把表格和前后的文字混在一起切。表格是结构化数据，应该整张保留为一个chunk，并且在前面加一句话描述这个表格是关于什么的。

原则二：chunk不是越小越好
很多教程告诉你chunk要小一点，256-512token比较好。这个建议在很多场景下是错的。

chunk太小，上下文信息丢失。用户问"这个产品的退货政策是什么"，答案分散在三个小chunk里——退货条件在chunk A，退货流程在chunk B，退货期限在chunk C。检索只返回了chunk A，答案就不完整。

我的经验：chunk大小应该以"一个chunk能独立回答一个问题"为标准。 如果一个chunk被单独拿出来，脱离上下文也能让人看懂它在说什么，那它的大小就是对的。

有些情况下这意味着一个chunk可以是1000-2000token，没关系。宁可大一点导致检索回来的内容有冗余，也比小到把答案切碎了强。

原则三：每个chunk要加元数据
光有chunk内容是不够的。每个chunk应该附带：

来源标记：来自哪个文档、哪个章节
上下文摘要：这个chunk在讲什么（可以用LLM自动生成一句话摘要）
层级信息：如果是分层文档，标注它属于哪个上级标题
这些元数据在检索时可以用来做过滤和排序，比如用户问的是"2024年的政策"，你可以先按年份过滤掉不相关的chunk，再做语义检索，精度会大幅提升。

三、Embedding选型：别用默认的，换一个可能效果翻倍
很多人搭RAG的时候，embedding模型就用了框架默认的那个，然后从来没换过。

embedding模型的选择对检索效果的影响，可能比你调一个月的chunk策略还大。

几个实用建议：

中文场景别用纯英文模型。 听起来像废话，但我见过太多人用默认的all-MiniLM-L6-v2（纯英文模型）来做中文RAG，效果当然差。换成BGE-M3、Jina-v2这类对中文优化过的模型，命中率能涨20-30个点。
如果你的文档是垂直领域的（法律、医疗、金融），考虑在领域数据上微调embedding模型。 通用模型对"本金""利率""违约金"这些词的语义理解可能和你的业务理解不一致。微调不需要很多数据，几千条query-document对就够了。
做一个简单的评测。 准备50-100个测试问题和对应的标准答案chunk，用不同的embedding模型分别跑一遍，看Top-5召回率哪个高。这比凭感觉选型靠谱一万倍。
四、别只做向量检索，加一路关键词检索
这是提升最明显、成本最低的一个优化。

纯向量检索有一个天然的弱点：它不擅长精确匹配。

用户问"订单号202401150032的状态"，向量检索会把"202401150032"理解为一个语义模糊的数字序列，可能返回一堆和订单状态相关但完全不是这个订单号的结果。

解法很简单：混合检索（Hybrid Search）。 同时跑一路向量检索和一路BM25关键词检索，把两路的结果做融合排序（一般用RRF，即倒数排序融合）。


向量检索擅长语义理解——"退货政策"能匹配到"商品退换规则"。

关键词检索擅长精确匹配——"202401150032"就是要精确找到这个字符串。

两路互补，效果比单路好很多。大部分向量数据库（Weaviate、Milvus、Qdrant）都原生支持混合检索，配置也就几行代码的事。

举一个特别有说服力的例子：Claude Code。

Claude Code是Anthropic做的AI编程工具，它在处理代码库的时候，几乎不用向量检索，主要靠关键词搜索（grep/ripgrep）。 这件事第一次听说的时候挺反直觉的——AI编程工具不是应该用最先进的语义检索吗？

但你仔细想想就会发现，代码这种东西，恰恰是最适合关键词检索的场景。

为什么？因为代码里的函数名、类名、变量名、包名，都是精确的字符串标识符。你想找getUserById这个函数，最准确的方式就是直接grep它，而不是把它embedding成一个向量然后做相似度搜索。后者反而可能给你返回一堆"相关但不是你要的"的结果——比如getUser、findUserById、fetchUserInfo，全是噪音。

代码的语法是确定的、没有歧义的。一个函数名就是一个函数名，它不像自然语言那样有"换种说法表达同一个意思"的问题。Golang只有25个关键字，Python也就30多个。这种环境下，向量检索的"语义泛化能力"反而成了缺点——它把本该精确匹配的东西模糊化了。

Anthropic团队公开说过他们的设计哲学：给AI一个grep + 让AI多看几次代码，比给它一个花哨的向量数据库管用得多。 他们甚至在Claude Code里直接给模型提供shell访问权限，让模型自己决定怎么搜代码——通常就是几次grep + 几次cat文件。

这个案例对RAG有两个启发：

第一，不要迷信"向量检索一定比关键词检索高级"。 工具的好坏取决于场景。代码场景下，关键词检索就是最优解。

第二，关键词检索在企业知识库里也比你想象的更重要。 你的知识库里有没有类似"代码"那样的"精确标识符"？产品型号、订单号、SKU、合同编号、零件编号、错误代码、API名称——这些全都是关键词检索的强项，向量检索碰到它们就抓瞎。

所以混合检索不是"为了保险加上的"，它是必需品。任何一个真实的业务知识库，里面都同时包含"自然语言描述"和"精确标识符"两类信息。你只用其中一种检索方式，必然会丢一半的命中。

五、检索之后加一步Rerank，性价比极高
检索返回了Top-20的结果，但这20个结果的排序可能不太对——真正最相关的可能排在第7位，而排在第1位的可能只是表面相似。

加一个Rerank模型，对Top-20的结果重新排序，让真正相关的排到前面。

Rerank模型和embedding模型不同。embedding是把query和document分别编码然后算相似度（双塔模型），速度快但精度有限。Rerank是把query和每个document拼在一起，让模型做交叉注意力（Cross-Encoder），精度高但只能处理少量候选。

所以工程上的最佳实践是：embedding做初筛（从10万条里选出20条），Rerank做精排（从20条里选出最好的5条）。

Cohere Rerank、BGE-Reranker、Jina Reranker都是不错的选择。如果用Cohere的API，一次请求几分钱，但效果提升非常明显。

六、Query改写：很多时候不是知识库的问题，是用户的问题
有一种情况：你的chunk分得很好，embedding也选对了，但检索效果还是差。

原因可能是——用户的提问方式太口语化、太模糊、太短，导致和知识库里的表述对不上。

比如用户问"怎么退"，知识库里写的是"商品退换货流程及注意事项"。语义上有关联，但embedding不一定能捕捉到，因为query太短了，信息量不够。

解法：在检索之前，用LLM对用户的query做一次改写。

几种常见的改写策略：

扩展改写：把"怎么退"改成"商品退货的具体流程和条件是什么"
多query检索（Multi-Query）：让LLM从不同角度生成3-5个变体query，分别检索，合并结果。比如"怎么退"可以变成"退货流程""退货条件""退货运费谁承担"
HyDE（假设性文档嵌入）：让LLM先假装回答一下用户的问题，生成一段"假设性答案"，然后用这段答案去做检索。因为假设性答案的表述方式和知识库里的表述方式更接近，检索效果会好很多
这几种方法都不需要改动知识库，只改检索前的一个步骤，但效果提升非常显著。

七、进阶：把"检索"本身当成一个智能Agent来做

讲到这里，前面六个方法都属于"优化RAG的零件"。但如果你做完这些发现还是不够好，下一步就要考虑一个更深的架构问题——

为什么RAG的检索环节一定是"被动"的？

传统RAG的逻辑是：用户发问 → 拿着这个问题去检索 → 把结果丢给LLM → 生成回答。整个检索过程是机械的、单步的、没有"思考"的。

但用户的问题往往很模糊。"什么手表"——你拿这四个字去做向量检索，能搜出什么？模型也不知道用户到底想问什么品牌、什么价位、什么款式。

我最近在一个客服场景的项目里在做的事情，就是把这个问题彻底换一种解法——用一个专门的"监管Agent"（Supervisor Agent）来负责检索决策。

简单说，架构变成了这样：

用户消息 → Supervisor Agent（理解上下文、生成精准检索查询、判断需要哪些类型的知识）→ 检索系统 → 把精筛过的知识 + 上下文摘要丢给Reply LLM → 生成回答
Supervisor做的事情包括：

第一，把模糊query改成精准query。 用户说"什么手表"，Supervisor结合这个会话的历史（比如这家店是个运动品牌），自动生成检索查询：["店铺手表类商品列表", "运动手表品牌款式"]。检索质量瞬间从"看天吃饭"变成"指哪打哪"。

第二，多维度并行检索。 用户说"我要赔偿前两天买的鞋"，Supervisor会拆解出三个维度的检索查询：["LP001运动鞋 赔偿政策", "物流异常 处理流程", "商品质量问题 赔偿标准"]。一次检索拿全所有相关知识，而不是只搜一次然后漏掉一半。

第三，会话摘要替代原始历史。 这是个隐藏的杀手级优化。传统RAG把14条历史消息全塞进上下文，里面充斥着AI上一轮可能编造的内容——比如AI上一轮编了一个"99999元"的价格，下一轮Reply LLM看到上下文里有这个数字，就会把它当成事实强化一遍。Supervisor负责维护一个会话摘要——把14条原始消息压缩成300字符的核心信息，并且明确标注哪些是用户说的、哪些是事实、哪些是待确认的。Reply LLM拿到的就不再是一堆混乱的历史，而是一份"已经梳理过的案情简报"。

第四，工作流和问题解决判断。 Supervisor还能判断"用户的问题是不是已经解决了""要不要进入退款流程""要不要发结束语"。这些是传统RAG完全做不到的事情。

最关键的一点——Supervisor和Reply LLM是并行执行的，不会增加延迟。 Supervisor当前轮的输出供下一轮Reply LLM使用，Reply LLM当前轮用的是上一轮Supervisor的输出。整个系统的响应时间不变，但每一轮的质量都在被持续优化。

我在自己的项目里跑下来，这种架构带来的效果提升大概是这样的（粗略对比）：

知识检索命中率：从60%左右提升到85%以上
上下文token消耗：14条原始消息(~7000字符) → 摘要(~300字符)，降了20倍
AI编造问题：因为摘要会明确区分"事实vs猜测"，编造内容的传染问题大幅缓解
工作流触发准确率：从延迟一轮的关键词匹配，变成实时的语义判断
这个思路在业界有个正在火起来的名字——Agentic RAG。它的本质是：RAG不再是一个固定的流水线，而是由Agent动态决策的智能过程。

如果你的场景是简单的"文档问答"，前面六个方法基本够用。但如果你的场景是多轮对话、有工作流、有上下文依赖（比如客服、法律咨询、医疗问诊），强烈建议往Agentic RAG的方向走。前面六个方法是在调零件，Supervisor架构是在重新设计引擎。

八、最容易被忽略的一步：清洗你的数据
我知道这个建议很"不性感"，但它可能是所有优化里ROI最高的一个。

你的原始文档里有多少"垃圾"？

页眉页脚（"第X页""机密文件""公司名称"反复出现）
目录页（一堆标题和页码，没有实际内容）
格式转换残留（PDF转文本后的乱码、错位、表格炸裂）
重复内容（同一段话在多个文档中出现）
无意义的模板文字（"以下内容仅供参考""如有疑问请联系XXX"）
这些东西如果不清理，会被切成chunk、生成embedding、存进向量数据库，然后在检索时和真正有用的内容竞争排名。它们不会帮你找到答案，但会稀释真正答案的排名。

花一天时间写几个清洗脚本——去掉页眉页脚、删除目录页、处理表格、去重——这一天的投入可能比你折腾一个月的chunk策略更有效。

九、建立一个评测体系，不要靠"感觉"
最后一个建议，也是最重要的一个。

如果你没有一个量化的评测体系，你做的所有优化都是在盲调。

怎么建？不需要很复杂：

准备一组测试集：50-100个真实用户会问的问题 + 每个问题的标准答案chunk
定义指标：Top-1命中率、Top-5召回率、MRR（平均倒数排名）
每次做完一个优化，跑一遍测试集，看指标变了多少
有了这个体系之后，你才能知道：换embedding模型涨了多少？加Rerank涨了多少？改chunk策略涨了多少？哪个优化的性价比最高？

没有评测体系的RAG优化就像没有体重秤的减肥——你觉得自己瘦了，但实际可能根本没变化。

总结：优化优先级
如果你时间有限，按这个顺序来，性价比从高到低：

清洗数据 — 投入最小，效果最直接
换embedding模型 — 一行代码的改动，可能涨20+个点
加混合检索（BM25 + 向量） — 几行配置，立竿见影
优化分段策略 — 按语义切、保留结构、加元数据
加Rerank — 精排一步到位
Query改写 — 解决用户表述和知识库表述不匹配的问题
建评测体系 — 让后续所有优化都有据可依
进阶：Supervisor Agent / Agentic RAG — 当上面这些都做了还不够，是时候重新设计架构了
别试图一步到位。 从第1步做到第3步，大部分场景的命中率就能从60%跳到85%以上。剩下的优化是锦上添花。

最后说一句：RAG不是一个"搭完就不管"的东西。你的知识库在变、用户的提问方式在变、模型在迭代——RAG是一个需要持续维护的系统工程。接受这一点，心态会好很多。
