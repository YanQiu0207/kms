# 评审总览

**一句话结论**：方案框架正确、分层清晰，作为个人知识库 MVP 的设计蓝图**基本可以落地**，但存在若干**工程细节错误**和**中文场景缺失**，若不修正会在真实文档上出现明显的"能跑不能用"问题。

评分（满分 5 分）：

| 维度         | 评分    | 说明                                              |
| ------------ | ------- | ------------------------------------------------- |
| 架构合理性   | 4.5 / 5 | 分层、链路、模块划分都对                          |
| 检索设计     | 3 / 5   | 混合检索方向正确，但分数融合与中文分词处理有缺陷  |
| 回答可信性   | 4 / 5   | 引用 + 拒答思路完整，但缺少引用校验闭环           |
| 工程可行性   | 3.5 / 5 | 技术栈合理，但部分配置/模型名不可直接使用         |
| 可演进性     | 4 / 5   | 分阶段路线清晰，预留了 rerank / query rewrite     |
| 完整度       | 3.5 / 5 | 缺少评估指标、缓存、安全等工程要素                |

下面按"值得保留 / 必须修正 / 建议补充"三类展开。

## 一、值得保留的设计决策

这些决策是方案里最有价值的部分，应当延续：

1. **明确拒绝"整篇 Markdown 入库"**。这是绝大多数 RAG 翻车的第一原因，方案一开始就点明，方向正确。
2. **命令层与服务层解耦**（`/kms` 薄 CLI + 本地 `kms-api`）。对个人项目很克制，后续换前端成本低。
3. **把“问题扩写”降级为兜底策略**。这条判断非常正确，扩写作为主路径会放大噪声，这点很多开源 RAG 模板都做错了。
4. **拒答机制前置**。`abstained` / `confidence` 作为响应字段是认真对待可信性的表现。
5. **切块策略基于标题层级而非固定长度**，并保留 `title_path`。这对个人笔记类文档尤其重要，因为标题往往携带关键语义。
6. **分阶段实施路线**（MVP → 可用版 → 稳定版）清晰克制，符合个人项目迭代节奏。

## 二、必须修正的问题

这些问题会直接影响第一版的可用性，建议在编码前修正方案。

### 2.1 分数归一化方案不能直接沿用

原文（19.3）：

```text
final_score = lexical_score * 0.45 + semantic_score * 0.55
```

**问题**：SQLite FTS5 的 `bm25()` 排序函数返回的是 **负数**（值越小越相关，并非越大越相关），而向量相似度通常是 0~1 的余弦相似度。两者量纲、方向都不同，直接加权会得到毫无意义的分数。

**建议修正**：优先采用 **RRF（倒数排序融合）**：

```text
final_score(d) = Σ 1 / (k + rank_i(d))
```

其中 `k` 取 60 左右，`rank_i` 是该文档在第 i 路召回中的名次。RRF 不依赖分数量纲，对不同检索器鲁棒，是目前开源混合检索（如 Elasticsearch、Weaviate）的默认融合方式。

如果一定要用加权融合，必须先做 min-max 归一化且处理 BM25 的符号问题，方案里应明确写出归一化公式，而不是只给一个权重。

### 2.2 FTS5 对中文基本不可用，必须替换分词器

原文（17.2）：

```sql
tokenize='unicode61'
```

**问题**：`unicode61` 按 Unicode 类别切分，对中文基本退化为**按单字切分**，召回质量非常差。例如查询"混合检索"，会被切成"混 合 检 索"四个单字，导致 BM25 几乎无法区分相关文档。

**建议修正**（任选其一）：

- **简单可行**：在写入 FTS5 之前用 `jieba` 做中文分词，把分词结果以空格拼接后再入库，检索侧同样分词后检索。此方案零原生依赖。
- **更规范**：使用 `signal-fts5-extension` / `simple` 分词器（支持中文 + 拼音），或切换到 `Meilisearch` / `Elasticsearch` 这类自带中文分词的引擎。

这是整套方案里**最容易被忽略但破坏性最大**的细节，MVP 阶段必须解决。

### 2.3 示例配置中的模型名不成立

原文（23）：

```env
ANSWER_MODEL=gpt-5.4-mini
```

**问题**：截至 2026-04-14，我没有查到 OpenAI 发布过 `gpt-5.4-mini` 这个型号（此为事实性内容，未经官方文档确认，仅能说"据我所知没有"）。如果用户直接复用这组环境变量，会得到 404。

**建议修正**：示例中改成明确存在的占位型号，并标注"请按实际提供商替换"。例如：

```env
ANSWER_MODEL=gpt-4o-mini          # 或 claude-haiku-4-5 / qwen2.5-7b-instruct 等
EMBEDDING_MODEL=text-embedding-3-small   # 或 bge-m3 / bge-large-zh-v1.5 等
```

同时建议方案里专门加一小节"模型选型"，至少区分 API 模型与本地模型两条路径。

### 2.4 中文知识库的 embedding 首选不该是 `text-embedding-3-large`

原文（9、23）默认推荐 `text-embedding-3-large`。

**问题**：

- 中文语义任务上，BAAI 的 `bge-m3`、`bge-large-zh-v1.5` 在公开评测（C-MTEB）中的综合表现**优于**或**持平于** OpenAI 的大型 embedding，而且支持本地部署、零成本、隐私友好（此处为方向性结论，具体分数建议读者查 C-MTEB 最新榜单再决策）。
- 个人知识库往往含有私人笔记，送到 OpenAI 做 embedding 有隐私顾虑。

**建议修正**：MVP 阶段默认推荐 `bge-m3`（支持稠密 + 稀疏 + 多向量，刚好契合混合检索）或 `bge-large-zh-v1.5`，API 模型作为备选。

### 2.5 仅靠 prompt 约束无法实现"只基于资料回答"

方案在 2.4 已经点出这一点，但第 20 节给出的方案仍然主要是 prompt + 阈值，缺少**后置校验**。

**建议补充一层答案引用校验**：

1. 要求模型输出结构化引用（例如在答案末尾附 `[1][2]` 这样的标号）。
2. 服务端解析引用标号 → 找到对应 chunk → 用简单的 n-gram 重叠或 embedding 相似度校验"答案中的关键句"是否真的能在被引用 chunk 中找到依据。
3. 如果校验失败，降级为拒答或标注"可能幻觉"。

这是从"看起来像基于资料"到"真正基于资料"的关键一步。

## 三、建议补充但 MVP 可延后

以下是方案里不够充分、但不影响第一版跑起来的点，建议作为 TODO 记入迭代计划。

### 3.1 向量库选择理由不清

方案给出 `Chroma / Qdrant` 二选一，但没讲区别。对个人知识库场景，可以补充一条：

- **`sqlite-vec`**（SQLite 向量扩展）：和现有 SQLite 共用一个文件，**零额外进程**，部署最简。个人规模（<10 万 chunk）性能足够。
- **Chroma**：文件级、Python 友好，MVP 最快。
- **Qdrant**：独立服务，未来多客户端时更省心。

对个人项目，推荐 `sqlite-vec` 作为"更克制"的选项；`Chroma` 作为"最易上手"的选项。

### 3.2 重排对中文语义检索收益很大，应在“可用版”而非“稳定版”引入

原文把 rerank 放到第三阶段（稳定版）。

**建议调整**：`bge-reranker-v2-m3` 这类 cross-encoder 对中文语义检索的 top5 质量提升非常显著，且接入成本极低（一个 HTTP 调用或本地模型）。可以把 rerank 提前到"可用版"，作为 MVP 之后的第一件要做的事。

### 3.3 增量索引的粒度问题

方案靠 `file_hash` 判断是否更新整篇。

**问题**：一旦切块策略（chunker）或 embedding 模型变更，所有旧 chunk 都需要重建，但 `file_hash` 无法感知。

**建议补充**：在 `documents` 或 `chunks` 表里加两列：

- `chunker_version`
- `embedding_model`

重建时比对这两列，只要版本不一致就强制重建，避免"改了切块逻辑但忘了 reindex"的静默失败。

### 3.4 缺少评估指标

第 24 节的测试策略是**定性**的（"能不能命中"），没有量化指标。

**建议补充**：准备一份 20~50 条的 `(query, expected_chunk_ids)` 评测集，跑如下指标：

- `Recall@k`（正确 chunk 是否出现在 top-k 中）
- `MRR`（正确 chunk 的平均倒数排名）
- `Abstain Precision`（拒答时是否真的应该拒答）

每次调权重、换模型、改切块，都跑一遍对比。这是避免"凭感觉调参"的最小代价。

### 3.5 缺少缓存层

个人使用场景下，同样的问题会反复问，建议最少加两层缓存：

- **向量缓存**：按 `sha1(text + model)` 作为 key，避免重复计算 embedding（尤其全量重建索引时）。
- **Query → Hits 缓存**：短 TTL（例如 5 分钟）即可，调试期体验差异明显。

### 3.6 Claude Code 技能集成方式没写清

方案说"`/kms` 作为命令入口"，但没说**具体怎么集成**。当前 Claude Code 有至少三种可选方案（此处为我对 Claude Code 生态的当前认知，具体接口请以官方文档为准）：

- 用 `.claude/skills/kms/` 目录做成一个本地技能，里面的 `SKILL.md` 指导 Claude 用 Bash 调用 `curl http://localhost:xxxx/ask`。
- 用 `.mcp.json` 注册一个 MCP server，把 `/index /search /ask /doctor` 暴露为 MCP 工具，这样不仅 `/kms` 能用，Claude 在日常对话里也能自动调用。
- 纯斜杠命令：`.claude/commands/kms.md`，但能力最弱。

建议在方案里显式选一种，并说明理由。对个人长期使用，**MCP server** 方案长远收益最大——知识库能力会自动出现在所有 Claude 对话里，而不是仅在显式 `/kms ask` 时。

### 3.7 安全 / 多端接入没提

本地 API 即便只跑在 `127.0.0.1`，在多端接入后（Obsidian 插件、网页）也会涉及：

- CORS 配置
- 简单的 bearer token 鉴权（防止本机其他进程乱调）
- 日志脱敏（问答内容可能含私人信息）

MVP 可以不做，但应写入"稳定版"TODO。

## 四、文档本身的一点排版反馈

按中文 Markdown 排版规范（`md-zh`）粗扫原文，以下几处可以在下一版里统一：

- 代码块外部的行内代码与中文紧贴时偶有缺空格，例如 **"将 chunk 同时写入全文索引和向量索引"** 这类是对的；但部分地方的 `SQLite FTS5` 前后空格不统一（少数段落缺空格）。
- 第 22 节伪代码里 `embed(chunk.text)` 后紧跟 `save_chunk_to_vector_store(chunk, emb)`，逻辑正确，但建议显式说明"embedding 失败时的重试与回滚策略"。
- 第 15 节 JSON 示例里的 `elapsed_ms` 没在文中定义单位（毫秒），对读者友好起见建议注明。

这些都属于打磨级别，不影响落地。

## 五、综合建议

如果把原方案视作"蓝图 V1"，建议按如下顺序**小改一版**再进入编码：

1. **立刻修正**（blocker）：
    - 分数融合改为 RRF（或明确写出归一化方案）。
    - FTS5 中文分词方案明确写出来（jieba 预分词或扩展分词器）。
    - env 示例里的模型名换成真实存在的。
2. **MVP 编码前补齐**（会省大量后续返工）：
    - 选定 embedding：优先 `bge-m3`。
    - 选定向量库：`sqlite-vec`（最克制） 或 `Chroma`（最易上手），说明理由。
    - 选定 Claude Code 集成方式：推荐 **MCP server**。
    - 增量索引加上 `chunker_version` / `embedding_model` 列。
3. **MVP 完成后立刻补**：
    - 准备 20~50 条评测集，跑 `Recall@k` / `MRR`。
    - 引入 `bge-reranker-v2-m3` 精排。
    - 实现"引用回校验"闭环。

完成这三步后，你得到的就不再是一个"能跑的 RAG Demo"，而是一个**可以长期作为个人知识库使用**的系统。

## 六、一页速记（给实现者）

- **融合**：用 RRF，不要拍脑袋加权。
- **中文**：FTS5 必须预分词，不要依赖 `unicode61`。
- **向量模型**：`bge-m3` 为首选，隐私更友好且中文能力强。
- **Rerank**：别等"稳定版"，MVP 跑通后立刻加。
- **引用**：不仅要输出引用，还要**后置校验**引用是否真的支撑答案。
- **集成**：`/kms` 用 MCP server 最值，能在所有对话里自动调用。
- **评估**：上线第一天就要有评测集，不然调参就是玄学。

以上。这份方案作为"个人 RAG 起手式"，基础款已经很稳；把上面几个必修点补上，它就能作为你未来一年的主力知识库跑下去。
