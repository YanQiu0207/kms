# 优化方案通用模板

## 使用说明

本文档用于承载需要分阶段推进、可量化验证、并且必须控制误伤风险的优化类任务。

适用场景包括但不限于：

- 检索优化
- 索引与数据清洗
- 排序与召回优化
- 协议兼容与实时链路优化
- 观测性、回归与稳定性增强

不适用场景：

- 一次性的小修小补
- 不需要前后对比验证的纯文案修改
- 只有单点实现、没有阶段性风险控制需求的任务

## 目标

本次优化不以“技术上更先进”或“看起来更好”为目标，而以：

- 核心业务指标提升
- 风险可量化
- 改动可回滚
- 收益可复现验证

为目标。

建议把目标区拆成下面三段分别填写，避免把收益、风险和失败红线混在一起：

本次优化重点提升：

- 指标 1
- 指标 2
- 指标 3

本次优化重点控制以下风险：

- 风险 1
- 风险 2
- 风险 3

本次优化不能接受以下副作用：

- 红线 1
- 红线 2
- 红线 3

本阶段优先解决：

- 问题 1
- 问题 2
- 问题 3

## 范围

本阶段覆盖：

- 模块 / 路径 / 服务 A
- 模块 / 路径 / 服务 B
- 配置、测试、文档

本阶段不覆盖：

- 与本次目标弱相关的旁路问题
- 大规模架构重写
- 宿主业务逻辑重构
- 顺手型优化

## 现状

当前主链路：

`input -> process-a -> process-b -> output`

现状核对：

- 组件 A 当前职责
- 组件 B 当前瓶颈
- 组件 C 当前风险

因此，本阶段的关键不是重写整个链路，而是：

在 `<关键杠杆点>` 插入或改造 `<能力层 / 适配层 / 清洗层 / 协议层>`。

## 设计原则

1. 先保守，再增强
   第一版只做低风险、高确定性的动作。

2. 先做对照，再默认开启
   所有优化都必须支持开关、灰度或回滚。

3. 为业务结果服务，不为技术动作服务
   不为了“更优雅”牺牲稳定性、追溯性和可解释性。

4. 通用规则优先，特例规则后置
   先抽共性，再补 source-specific 或 case-specific 规则。

5. 高风险动作先标记，再决定是否自动执行
   优先通过统计、抽样和人工确认降低误伤。

## 阶段计划

## Stage 0：基线与对照体系

### 目标

建立可靠对照组，确保后续每一步都能量化收益和风险。

### 交付物

- 固定样本集
- benchmark / 回归集
- 指标定义
- 输出目录与命名约定
- 对照报告模板

### 动作

1. 固定输入样本
2. 固定测试问题 / 请求集
3. 固定比较指标
4. 固定输出位置

### 验收

- 可以稳定复跑
- 可以产出 baseline 结果
- 后续版本可以与 baseline 一对一比较

## Stage 1：低风险优化

### 目标

先落最低风险、最高确定性的改动，不改变主要语义边界或核心协议语义。

### 交付物

- 核心能力最小实现
- 配置开关
- 基础测试
- 回滚路径

### 动作

1. 增加能力层或适配层
2. 只实现低风险规则
3. 补充必要 metadata / 标记 / 统计
4. 配置化控制默认行为

### 验收

- 核心指标不回退
- 风险指标不明显恶化
- 支持关闭和回滚

## Stage 2：结构化增强

### 目标

处理更复杂但收益更高的场景，在收益与可解释性之间保持平衡。

### 交付物

- 增强解析或增强策略
- 复杂场景处理规则
- 风险标记机制
- 局部覆盖能力

### 动作

1. 引入增强解析或结构识别
2. 保留结构化信息，避免粗暴压平
3. 对高风险动作先打标、先统计
4. 支持按来源或场景做局部覆盖

### 验收

- 复杂场景指标稳定提升
- 可解释性保持
- 能输出典型收益样例

## Stage 3：高级抑制与排序优化

### 目标

处理重复、竞争、排序稀释、误召回等高级问题。

### 交付物

- 离线分析结果
- 运行时策略
- 保守版默认策略

### 动作

1. 先做离线统计与人工抽样确认
2. 再决定线上策略
3. 优先采用保守方案

### 验收

- 排序质量提升
- 不出现明显召回下降
- 没有大规模误伤

## 对比要求

每轮版本都必须输出以下对比：

1. benchmark 对比
   - overall
   - by_type
   - by_tag / by_case

2. 系统侧统计对比
   - 总量
   - 分布
   - 重复率 / 异常率
   - 典型异常样本

3. case diff
   - 至少抽若干条代表性 case
   - 对比命中、排序、输出、误答 / 拒答变化

4. 样本 diff
   - 输入
   - 中间结果
   - 最终结果
   - 为什么这条规则有益且没有伤主语义

## 放行与拦截规则

默认放行条件：

- 风险指标不升高
- 核心收益指标不出现明显回退
- 至少 1 个核心指标有实质改善
- 没有发现大规模误伤

默认拦截条件：

- 风险指标明显恶化
- 核心召回 / 准确率明显下降
- 样本中出现关键信息丢失
- 规则无法关闭、灰度或回滚

## 目标填写参考

如果你写的是 `mykms` 这类 RAG / 数据清洗 / 检索优化方案，可以优先从下面三组里挑：

本次优化重点提升：

- `mrr`
- `recall_at_k`
- `evidence_hit_rate`
- `evidence_source_recall`
- `expected_term_coverage`

本次优化重点控制以下风险：

- 清洗过度导致有效证据被误删
- 结构化内容处理不当导致可检索性下降
- 去重或降权策略误伤真实不同的知识片段
- 增量索引稳定性下降
- baseline 与 case diff 缺失导致回归不可见

本次优化不能接受以下副作用：

- `false_answer_rate` 明显升高
- `recall_at_k` 明显下降
- 关键信息丢失或来源追溯能力被破坏
- 改动无法通过配置关闭、灰度或回滚

## 风险

1. 优化过度，误伤有效信息
2. 输入分布不一致，规则泛化差
3. 增强解析引入额外复杂度
4. 排序或去重策略误伤边界 case
5. 新依赖提高环境与维护成本

## Review 关注点

Review A 重点看：

- 方案是否破坏原始语义
- 指标与收益是否真实
- 对比方法是否可复现

Review B 重点看：

- 工程接入是否稳定
- 开关、回滚、测试是否完整
- 新依赖是否必要

## 验收

- 阶段划分明确
- 指标与放行条件明确
- 回滚和开关方案明确
- 默认按 Stage 0 -> Stage 1 -> Stage 2 -> Stage 3 顺序推进
- 每阶段上线前必须产出前后对比结果
