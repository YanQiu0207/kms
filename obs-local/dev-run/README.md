# obs-local 开发运行手册

本目录用于管理 `obs-local` 的开发执行、并行 agent 协作、阶段评审、bug 跟踪与修复闭环。

主会话是总协调者，负责：

- 拆阶段
- 分配 agent
- 控制写入边界
- 合并代码
- 决定阶段是否通过
- 维护本地 md 台账

子 agent 负责在明确边界内并行开发或评审，不单独决定阶段放行。

## 强制规则

1. 开发必须按阶段推进。
2. 只要写入范围不冲突，就应使用多个 agent 并行开发。
3. 每个阶段的进度、审核、bug、修复，都必须记录在本目录的 md 文件里。
4. 每个阶段开发完成后，必须拉 2 个独立 agent 做代码审核。
5. 审核意见里，凡是主会话认定为“接受”的问题，必须修完并记录验证结果。
6. 当前阶段未在 `stage-status.md` 中标记为“已通过”前，不得进入下一阶段。

## 文件说明

- `master-plan.md`：完整开发总计划
- `stage-status.md`：阶段状态板与放行闸门
- `agent-board.md`：agent 分工、写入范围、交接说明
- `progress.md`：执行进展日志
- `review-log.md`：每轮审核记录
- `bug-ledger.md`：已接受问题与缺陷台账
- `fix-log.md`：修复记录与修复后验证

## 阶段执行流程

1. 主会话从 `master-plan.md` 里选择当前阶段。
2. 主会话把阶段拆成可并行的工作流。
3. 在 `agent-board.md` 里登记 ownership。
4. 多个 agent 并行开发。
5. 主会话合并代码并做本地验证。
6. 冻结当前阶段代码范围。
7. 启动 2 个独立 review agent 做审核。
8. 在 `review-log.md` 中登记审核意见。
9. 将“接受的问题”写入 `bug-ledger.md`。
10. 修复后在 `fix-log.md` 记录修改与验证结果。
11. 主会话确认问题已关闭，并在 `stage-status.md` 中标记通过。

## 审核闸门

每个阶段放行前必须同时满足：

- 本阶段实现完成
- 本阶段本地验证已记录
- Review A 完成
- Review B 完成
- 已接受问题全部修复
- 修复后的验证结果已记录
- 主会话明确写入“通过”
