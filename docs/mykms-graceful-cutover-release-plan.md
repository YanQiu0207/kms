# mykms 平滑切换发布方案

本文是 `mykms` 的平滑切换发布设计稿，**只做方案，不做实施**。

目标不是“模块热更”，而是：**在不让用户明显受影响的前提下，完成新版本上线、切流和回滚。**

## 1. 目标

当前希望解决的问题是：

- 修一个后端 bug，不想手工停服务再起服务
- 不希望因为一次发布把正在使用的用户直接打断
- 不希望走 Python 进程内模块热替换这条高风险路线

本方案的目标是：

- 对外保持一个稳定入口
- 新版本先在备用实例完成启动、健康检查和预热
- 确认可用后再切流
- 切流后让旧实例优雅退出，而不是直接掐断
- 一旦发现问题，可以快速回滚到旧实例

## 2. 非目标

本文明确不做这些事情：

- 不做 Python 进程内模块热替换
- 不做“改完代码自动替换老对象”的 runtime patch
- 不追求完全零连接波动
- 不在本方案里同时改 `obs-local`

原因很简单：

- `mykms` 在启动期会创建长生命周期对象，例如 [app/main.py](/E:/github/mykms/app/main.py) 里的 `IndexingService` 和 `QueryService`
- 这些对象背后有缓存、模型资源、数据库连接、vector store 句柄
- 如果只 reload 模块，而不完整重建进程和对象，极易出现新旧代码混跑、状态脏化、资源泄漏

所以：**开发期可以做 `--reload`，发布期必须做“新旧实例切换”，而不是进程内热更。**

## 3. 当前现状

基于仓库当前实现，相关事实如下：

- `mykms` 目前固定监听 `127.0.0.1:49153`
  - 见 [app/config.py](/E:/github/mykms/app/config.py)
- 当前启动脚本是后台直接拉起单实例
  - [scripts/start_kms.py](/E:/github/mykms/scripts/start_kms.py)
- 当前停止脚本是按 pid / 端口去优雅停进程
  - [scripts/stop_kms.py](/E:/github/mykms/scripts/stop_kms.py)
- 当前已经有健康检查：
  - `GET /health`
  - 见 [app/main.py](/E:/github/mykms/app/main.py)
- 当前配置里有 `warmup_on_startup`
  - 见 [app/config.py](/E:/github/mykms/app/config.py)

这意味着：

- 现在的架构是“客户端直连单实例”
- 这种结构下，只要重启当前进程，用户就一定会遇到短暂不可用
- 如果不先引入稳定入口层，就没有办法做到真正的平滑切换

## 4. 核心设计

### 4.1 总体思路

采用 **蓝绿切换** 或 **主备切换** 模式：

- 对外保留一个稳定入口：`127.0.0.1:49153`
- 真实 `kms-api` 实例不再直接暴露给用户
- 新旧版本运行在两个内部端口上
- 稳定入口层负责把流量转发到当前活动实例

推荐端口布局：

- 稳定入口：`49153`
- 蓝实例：`49163`
- 绿实例：`49164`

切流时，**用户永远访问 `49153`**，内部再决定当前代理到蓝还是绿。

### 4.2 组件划分

建议把发布形态拆成三层：

1. 稳定入口层
- 职责：
  - 占住 `49153`
  - 只做反向代理 / 转发
  - 保存当前活动 upstream

2. `kms-api` 实例层
- 蓝实例：
  - 旧版本或当前线上版本
- 绿实例：
  - 新版本候选

3. 发布控制层
- 职责：
  - 启动候选实例
  - 做健康检查和预热确认
  - 执行切流
  - 触发旧实例 drain 和停止
  - 出问题时执行回滚

## 5. 为什么必须先有稳定入口

这是整套方案里最关键的一点。

如果没有稳定入口，而是仍然让用户直连 `49153` 上的真实 `kms-api` 实例，那么：

- 新版本上线必须抢占同一个端口
- 抢占端口前必须停掉旧版本
- 这段时间内用户必然看到错误或连接失败

所以平滑切换的前提不是“后端会不会自动 reload”，而是：

**用户访问的地址必须先和具体进程解耦。**

换句话说：

- 没有代理层：只能停旧起新
- 有代理层：才能先起新、后切流、再停旧

## 6. 建议的发布流程

### Phase 0：识别当前活动实例

发布控制器先读取当前活动状态，例如：

- 当前活动实例：`blue`
- 当前活动端口：`49163`
- 当前稳定入口 upstream：`127.0.0.1:49163`

这个状态可以单独存成一个本地状态文件。

### Phase 1：拉起候选实例

假设当前线上是蓝实例，则本轮拉起绿实例：

- 新版本启动到 `127.0.0.1:49164`
- 不接用户流量
- 只接受内部探活与验证

启动后执行：

1. `GET /health`
2. 必要的预热确认
3. 可选的冒烟检查：
   - `/stats`
   - 简单 `/search`
   - 简单 `/ask`

### Phase 2：就绪校验

候选实例必须满足这些条件才允许切流：

- 进程存活
- 监听端口成功
- `/health` 通过
- 模型预热完成，或至少达到可服务状态
- 关键依赖可用：
  - SQLite
  - Chroma
  - embedding / reranker

注意：

当前 `/health` 还偏轻量。正式实施时，更合理的是增加一个更严格的 `readiness` 概念，而不是只看当前浅探活。

### Phase 3：切流

切流动作本质上是：

- 把稳定入口的 upstream
  - 从 `127.0.0.1:49163`
  - 切到 `127.0.0.1:49164`

理想要求：

- 切换动作原子化
- 新请求立即进入新实例
- 老请求仍允许在旧实例上自然完成

### Phase 4：旧实例 drain

切流后，不应立刻粗暴杀掉旧实例。

建议进入 drain 状态：

- 不再接新请求
- 保留一个有限的等待窗口
- 让已在处理中的请求完成

drain 结束后，再调用现有停止逻辑或其升级版，优雅停掉旧实例。

### Phase 5：发布完成

状态更新为：

- 当前活动实例：`green`
- 当前备用实例：`blue`
- 当前稳定入口 upstream：`127.0.0.1:49164`

## 7. 回滚流程

回滚必须和切流一样简单，不能依赖手工排查半天。

推荐回滚方式：

1. 如果新实例切流后发现异常：
- 立刻把稳定入口 upstream 切回旧实例

2. 旧实例如果仍在 drain 窗口内：
- 直接恢复为活动实例

3. 如果旧实例已经停掉：
- 重新拉起旧版本到备用端口
- 验证通过后再切回

回滚必须具备两个前提：

- 切流前旧实例不能过早退出
- 发布控制层必须记录“上一版本在哪个端口、哪个版本号”

## 8. 入口层选型建议

这里有三种路线。

### 方案 A：Nginx / Caddy / HAProxy

优点：

- 职责清晰
- 天然适合反向代理和切流
- 平滑切换语义成熟

缺点：

- Windows 本地维护成本更高
- 需要额外安装和运维一个代理组件

### 方案 B：仓库内自带一个轻量反向代理进程

优点：

- 全部逻辑都在仓库内
- Windows 本地环境更容易统一
- 可以和现有脚本集成

缺点：

- 需要自己实现：
  - upstream 状态管理
  - 原子切换
  - drain 控制
  - 错误处理

### 方案 C：端口占用 + 系统层转发

优点：

- 组件最少

缺点：

- 可维护性差
- 难做优雅切换
- 不建议作为正式方案

当前更推荐的方向是：

- **如果要尽快形成可控能力，优先方案 B**
- **如果后面要走更正式的部署形态，再评估方案 A**

原因：

- 这个项目当前本身已经有较多本地脚本和 Windows 场景约束
- 先做仓库内自带的轻量入口层，更符合当前工程风格

## 9. 对现有脚本体系的影响

正式实施时，现有脚本不该被直接推翻，而是要分角色演进。

### 当前脚本

- [scripts/start_kms.py](/E:/github/mykms/scripts/start_kms.py)
  - 现在是“启动唯一线上实例”
- [scripts/stop_kms.py](/E:/github/mykms/scripts/stop_kms.py)
  - 现在是“停止唯一线上实例”

### 未来角色

建议演进为：

- `start_kms.py`
  - 变成“启动指定槽位实例”
  - 例如：
    - `--slot blue`
    - `--slot green`
    - `--port 49163/49164`

- `stop_kms.py`
  - 支持按槽位停实例
  - 支持 drain timeout

- 新增发布控制脚本
  - 例如：
    - `deploy_kms.py`
    - `switch_kms.py`
    - `rollback_kms.py`

- 新增入口层启动脚本
  - 例如：
    - `start_kms_gateway.py`

## 10. 需要补的能力

本方案实施前，建议先补这些能力。

### 10.1 readiness 概念

当前只有 `/health`，但平滑切换更需要：

- “进程活着” 与 “可以接流量” 分开

建议后续补：

- `/ready`
  或
- 在 `/health` 里扩出更严格的 readiness 字段

### 10.2 drain 状态

切流后旧实例要能进入“只出不进”的状态。

这通常需要：

- 实例知道自己是否处于 draining
- draining 时拒绝新请求或由入口层不再转发新请求

### 10.3 发布状态文件

需要一个本地状态文件记录：

- 当前活动槽位
- 当前备用槽位
- upstream 端口
- 最近一次发布版本
- 最近一次回滚版本

### 10.4 预热确认

如果后续把 `warmup_on_startup` 用在正式发布链路里，就必须明确：

- 切流前一定要等预热结束
- 不能让用户请求去触发第一轮昂贵初始化

## 11. 推荐实施顺序

如果后续要真正落地，建议分四步做。

### Stage 0：设计确认

先拍板这些问题：

- 稳定入口选型：
  - 自带轻量代理
  - 还是外部 Nginx / Caddy
- 端口规划
- readiness 语义
- drain 策略
- 回滚保留窗口

### Stage 1：单机双实例能力

先让 `mykms` 能稳定跑：

- 蓝实例
- 绿实例

但还不切流。

### Stage 2：稳定入口与切流

加入稳定入口层，打通：

- 启动候选实例
- 健康检查
- 切 upstream

### Stage 3：drain 与回滚

收口：

- 旧实例优雅退出
- 快速回滚
- 发布状态台账

## 12. 风险与边界

### 风险 1：当前 `/health` 太轻，不足以代表可接流量

如果只看浅健康检查，可能出现：

- `/health` 是绿的
- 但首个真实 `/ask` 仍然卡住或失败

### 风险 2：模型预热和资源初始化耗时

如果新实例切流前没完成预热，用户仍会感觉“虽然没断，但第一批请求明显变慢”。

### 风险 3：旧实例长请求拖住 drain

如果某些请求耗时很长，drain 窗口需要明确上限，否则旧版本长期不退出。

### 风险 4：当前客户端可能直接写死 `49153`

这本身不是问题，但意味着：

- 入口层必须先接管 `49153`
- 后端实例不能继续直接占这个端口

## 13. 最终建议

对 `mykms` 来说，真正合理的方向不是：

- Python 模块热更

而是：

- **稳定入口 + 蓝绿实例 + 健康检查 + 切流 + drain + 回滚**

一句话总结：

**开发期用 `--reload` 提升改代码效率，发布期用平滑切换保证用户尽量无感。**

## 14. 当前结论

当前建议是：

1. 先不要做模块热替换
2. 后续如需真正减少用户感知中断，优先立项“平滑切换发布”
3. 本文作为后续实施版的前置设计稿，当前不直接改代码
