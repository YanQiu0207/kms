# Vue 3 UI Framework Selection

## 结论

如果目标是延续当前 `obs-local` 这套“高级控制台 / 深色运行态观测台”风格，在 `Vue 3` 下最适合的开源方向是：

- `Reka UI`

不是因为其它框架做不到，而是因为它最适合在不牺牲视觉自主权的前提下，承接长期组件化建设。

## 选择标准

这次选型的前提不是“最快搭后台”，而是：

- 保住当前这套高级控制台气质
- 不重复手搓复杂交互底座
- 后续可以继续扩展 tooltip、popover、menu、drawer、combobox、dialog 等能力
- 保留完整的视觉定制权

## 为什么是 Reka UI

### 1. Vue 原生、偏底座层

`Reka UI` 是面向 Vue 的无样式基础组件体系，更适合作为设计系统底座，而不是直接输出一套强视觉成品后台。

这和当前项目的目标一致：

- 交互行为借成熟实现
- 视觉语言继续自定义

### 2. 更适合维持当前风格

当前 `obs-local` 的界面特点是：

- 深色控制台
- 青蓝冷光
- 玻璃感渐变
- pill / badge / segmented control 风格
- 非标准企业后台模板感

这类风格更适合建立在无样式 primitives 上，而不是建立在默认视觉很强的现成企业组件库上。

### 3. 长期维护成本更低

如果继续完全自己写，后续会持续重复处理：

- 键盘导航
- focus 管理
- 弹层定位
- 关闭逻辑
- 可访问性
- 组合式状态机

`Reka UI` 更适合把这些基础交互成本外包掉，同时不强行接管页面视觉。

## 为什么不是直接用 Ant / Arco

不是因为：

- `Ant Design Vue` 做不到
- `Arco Design Vue` 做不到

而是因为它们默认更偏：

- 标准企业后台
- 成品型组件库
- 自带强视觉语言

如果直接使用默认样式，界面很容易回退到“常规后台感”，与当前这套控制台风格冲突。

它们当然可以深度定制，但代价通常是：

- 持续与默认皮肤打架
- 大量覆写 token 和组件样式
- 局部仍残留组件库味道

所以对当前项目来说，它们不是最优先选择。

## 为什么不是继续全手写

全手写不是不行，但长期不是最优：

- 小范围时速度可控
- 一旦控件种类增多，交互与可访问性成本会迅速累积
- 后续 tooltip、popover、menu、dialog、combobox、listbox、drawer 会不断重复造轮子

因此长期更合理的方式是：

- 交互底座用开源
- 视觉层继续自定义

## 推荐策略

当前建议策略：

1. 保留“自定义视觉语言”
2. 优先采用 `headless / unstyled primitives`
3. 若后续统一迁移，优先考虑 `Reka UI`
4. 不直接把重型企业 UI 框架默认皮肤引入控制台页面

## 当前状态

当前仓库还没有正式迁移到 `Reka UI`。

现状是：

- 已先落地 headless 交互方向
- 当前页面控件已去掉原生浏览器下拉
- 后续若继续演进组件体系，`Reka UI` 是优先候选

## 一句话结论

在 `Vue 3` 下，如果目标是做出并长期维持当前这套高级控制台风格，最合适的不是“直接套现成后台框架”，而是：

- `Reka UI` 这类无样式交互底座 + 自定义视觉系统
