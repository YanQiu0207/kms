# Frontend Console Guidelines

适用于控制台、观测台、仪表盘、实时运行态页面。

## 1. 目标

- 默认目标不是“能用”，而是“专业、克制、有产品感”。
- 页面应优先呈现统一的视觉语言，不接受局部控件掉档。

## 2. 必须

- 视觉层次统一：
  - 品牌区、状态区、统计卡、列表区、详情区共享同一套色阶、边框强度、阴影和圆角语言。
- 少量离散模式切换使用显式状态控件：
  - 优先 `segmented control`、`pill tabs`、`mode switch`
  - 不默认使用原生 `select`
- 国际化使用稳定 key：
  - 代码内部保存英文语义 key
  - UI 文案统一通过翻译层渲染
  - 支持后端默认语言、前端手动切换、本地覆盖
- 产品文案与技术标识分层：
  - 标题、按钮、空态、说明、状态标签走翻译层
  - `/ask`、`query.plan.fetch`、`request_id`、`span_name` 保留原值
- 内部状态传递 key，不传递自然语言：
  - store、SSE、轮询状态机中传播稳定 key 或枚举
- 窗口敏感数据以当前窗口 API 为准：
  - 如果 SSE 缺窗口维度，只能作为“失效通知”，不能直接覆盖页面数据

## 3. 应当

- 状态类元素使用同系胶囊或 badge 语言。
- 同一行控件保持一致的尺寸、密度和对齐节奏。
- 技术事件名可用“双层信息”展示：
  - 语义标签
  - 原始技术值
- 微动效应服务于层次和状态表达，只使用克制的入场、浮动、呼吸和高亮过渡。
- Tooltip 用于解释状态、术语和缩写，不用于补救混乱的信息架构。

## 4. 禁止

- 把原生系统控件样式直接暴露到成品界面。
- 在业务状态和控制流里硬编码中文或英文展示文案。
- 为了中文化而翻掉真实技术标识，降低排障能力。
- 在未确认真实协议前，先加自动重连、定时检测、自愈兜底。
- 遇到“界面没刷新”时，跳过分层排查。

## 5. 排查顺序

出现“界面没刷新”时，按顺序确认：

1. 生产者是否写出新事件
2. 聚合或后端 API 是否已看到新数据
3. 实时流是否真的发出新事件
4. 前端是否按真实 wire payload 正确消费

只有在协议和消费逻辑都确认正确后，才进入重连、自愈和保活层。
