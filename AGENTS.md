# AGENTS

本文件作为仓库内的全局补充规则，和外部协作规范一起生效。

## 协议与流式接口

- 任何 HTTP、SSE、日志、批处理事件的协议改动，都必须以真实 wire payload 为准，不能只根据类型名、变量名或本地抽象推断。
- 生产者与消费者之间只要存在 `batch`、`wrapper`、`payload`、`envelope` 这类嵌套层，修改任一侧后都必须核对一次真实报文样例。
- 前后端联动改动必须至少覆盖两类回归：
  - 单条事件直推
  - 批量事件封装后再消费
- 如果前端通过 dev proxy 消费后端流，还必须至少验证一次：
  - 直连后端接口
  - 经代理后的同一接口

## 排障顺序

- 当界面表现为“没有实时更新”时，先按下面顺序定位，不要先补自动重连或定时兜底：
  1. 确认生产者是否写出新事件
  2. 确认聚合/后端 API 是否已看到新数据
  3. 确认实时流是否真的发出新事件
  4. 对照真实 wire payload 检查前端消费逻辑
- 只有在确认协议和消费逻辑都正确后，才进入连接保活、重连、自愈这类补救层。

## 前端与国际化

- 控制台、观测台、仪表盘、实时运行态页面，以及相关国际化/文案规则，统一遵循 [docs/frontend-console-guidelines.md](/E:/github/mykms/docs/frontend-console-guidelines.md)。

## Benchmark 与测试案例纪律

- 默认禁止为了让结果更好看而修改 benchmark、评测集、golden case、回归样例或测试断言口径。
- 功能开发与 benchmark 维护必须分开：
  - 不能一边改实现，一边放宽案例口径，再把收益算成功能改进。
  - 功能收益必须基于同口径数据对比。
- 如果测试案例本身确实有问题，可以改，但必须同时满足下面条件：
  1. 能给出充分、可复核的证据，证明原案例有误、歧义或与真实目标不一致。
  2. 变更理由必须明确记录到 `dev-run/issue-log.md` 或对应阶段台账。
  3. 案例修正必须作为独立维护事项处理，不能与功能收益混算。
  4. 修正后要明确说明它影响了哪些历史结果，以及为什么这是“口径纠偏”而不是“结果优化”。
- 当 benchmark 存在争议时，优先选择更保守、更不占便宜的口径；如果仍无法达成无争议结论，就先保留争议并单独立项，不要拿它给当前开发放行。
